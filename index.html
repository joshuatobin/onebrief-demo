03 · What We Built

Two problems the provisioning system couldn't solve.

The internal build system provisions — but it doesn't know what to build, and it can't manage what happens when builds fail. We built both layers around it.

Part 1 · Inflation
Telling the build system what to build
Turning the service-instance list into the artifact knowledge the build system needs.
  • Started as manual spreadsheet toil — slow, error-prone, no accountability
  • Drove an ADR and got the platform team to build their first-ever deep link into the system
  • Moved it into an agent-based workflow — inflation is now automated end to end
Part 2 · Coordination
Managing the deploy lifecycle
The provisioning system has no concept of it — thousands of deploy notifications, and what to do when a build fails.
  • The build system has no way to coordinate notifications or handle failures at scale
  • We inject deterministic workflows + Mastra agents to notify, catch failures, and intervene
  • Payoff: faster iteration, service-owner trust, executives watching timelines accelerate
04 · What's Worth Owning

Build the differentiator.
Buy the rest.

Agents collapsed the cost of creation, so the question is no longer build versus buy — it's what is actually worth owning.

GIVEN
Internal provisioning system + managed AWS (EKS, RDS, Vault)
Upstream and standard. I adapted to them rather than fighting to replace them — the leverage was never going to come from reinventing infra.
Traded: no control over the provisioning system; had to engineer around its constraints.
BUILD
Build coordination tooling + bounded agents (Mastra)
No off-the-shelf tool fit the coordination problem — 200+ apps, dependency ordering, compliance gates. Deterministic code by default; agents only at the judgment points, test-gated, never unsupervised on the apply path.
Traded: we own the maintenance, and must verify agent output by tests rather than trust it.
You can generate code with an agent. You can't generate a FedRAMP authorization or a vendor's uptime. Build what differentiates you; buy the commodity.

Front-loaded the platform: the coordination tooling and the first region landed in Q1–Q2, before scaling out.
Sequenced by dependency risk — the services that unblocked the most downstream apps went first. Dependencies live in code, so build order was derived, not guessed.
Bounded AI agents fixed ordering bugs and fed each fix back in, so every region went faster than the last.
AI was the accelerant — deterministic code, agents injected at the judgment points. Never AI slop.
06 · Key Decisions

Navigating the terrain

The hardest calls weren't technical — they were organizational and compliance constraints.

Compliance as a gate, not a walldriving alignment
A platform mandate forced manual provisioning, no API. I drove an ADR and earned a seat at the platform team's design table, convincing them to build their first-ever deep link into the system. Engineered through the constraint by aligning the team that owned it — not by going around them.
Incremental over rewritevalue before perfect code
Inherited spreadsheets and tribal knowledge. Chose layer-by-layer automation over a stalling rewrite — each step shipped value immediately while migrating people off manual work.
The bottleneck was reasoning, not peopleleverage over headcount
100+ engineers, no headcount coming. Invested in parallel-work architecture, async handoffs, and agent-assisted tooling that multiplied output — making what we had 10× more effective.
The organization isn't the enemy. It's the terrain.Every compliance gate I passed left a tooling artifact that made the next pass faster.
07 · What Broke

What broke — and the fix that compounds.

01
What broke
Dependency ordering wrong in main.tf — apply ran out of order, downstream services came up broken.
02
Blast radius
Caught during region bring-up, before customer impact. Platform held at 99.9%.
03
Detection
Apply failures surfaced during bring-up; bounded agents were watching for ordering faults.
04
Diagnosis
A missing dependency edge. Wrong assumption: that every team had declared its real deps.
05
Fix
Agents proposed corrected Terraform; a test gate approved it. Agents only at diagnosis — never unsupervised on apply.
The systemic fix

Every fix was appended to the living blueprint, so the next region couldn't repeat it. The failure class shrank with every region — the mechanism behind 120 days dropping to 45.

In an IL5 or JWICS environment, that same gate gains a human approver and a signed audit trail. The architecture stays — the gate adapts to the compliance terrain.
What I'd do differently today
Agents first
We proved the bottleneck was coordination, then built the agent layer. Today I'd stand up the deterministic-workflow-and-agent layer before region one — region one should have looked like region three.
Gate it up front
Make "declare your dependencies in code" a hard gate to entering the build from day one, instead of catching mis-declared deps when a region broke.
